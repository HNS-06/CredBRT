# Credential_BRT

**Terminal-based credential-stuffing & brute-force detection engine.**

A real-time risk-scoring tool for login/auth telemetry — built to catch not
just loud brute-force attacks, but distributed, low-and-slow credential
stuffing that simple rate limiters miss.

```
 CREDENTIAL_BRT   brute-force & credential-stuffing detection engine
 ─────────────────────────────────────────────────────────────────────
 live attempt stream          |  top offenders — risk ranking
 activity sparkline           |  recent alerts
```

## Why this is different from a basic rate limiter

Most brute-force protection looks at **one IP's** request rate. That misses
two very common real-world attack shapes:

| Attack shape | What it looks like | Why naive rate-limiting misses it |
|---|---|---|
| **Password spray** | 1 IP → many accounts, 1 try each | Per-account rate stays low |
| **Credential stuffing (distributed)** | many IPs → 1 account, 2–3 tries each | Per-IP rate stays low |
| **Low-and-slow** | failures trickle in over hours from dozens of IPs | Never crosses a short-window threshold |

Credential_BRT treats the **account** and the **IP** as first-class entities
scored independently, and fuses four detectors into one composite risk score:

1. **Velocity** — classic sliding-window fail-rate, per IP and per account
2. **Stuffing (fan-out/fan-in)** — graph-degree counting: 1 IP → N accounts, or 1 account → N IPs
3. **Low-and-slow** — EWMA of long-window failure "heat" + Shannon entropy of
   failures across source IPs (evenly-spread failures across many IPs is the
   signature of a patient distributed attacker)
4. **Behavioral fingerprinting** — user-agent rotation + timing-interval
   regularity (coefficient of variation) to separate scripted bots from humans

Composite score (0–100) → tiered **LOW / MEDIUM / HIGH / CRITICAL**, all
tunable in `config.toml`.

## Install

```bash
pip install -e .
```

Requires Python 3.10+. Dependencies: `click`, `rich`, `tomli`, `tomli_w`.

## Quick start

```bash
credbrt init                 # creates config.toml + credbrt.db
credbrt simulate demo        # injects synthetic normal + attack traffic
credbrt monitor              # launch the live dashboard
```

`simulate demo` does **not** attack anything — it fabricates login-attempt
records directly in your local database (like test fixtures) so you can
watch the detectors light up before wiring in real data.

## Feeding it real data

Point it at a JSONL log where each line is:

```json
{"ts": 1720000000.0, "ip": "1.2.3.4", "account": "alice", "success": false, "user_agent": "curl/8.0", "country": "US", "asn": "AS123"}
```

```bash
credbrt ingest auth.log.jsonl     # tail -f style, live scoring
credbrt replay historical.jsonl   # bulk-load for retrospective analysis
```

Any auth system — nginx/Apache log post-processor, app middleware, IdP audit
export — can be adapted to emit this shape.

## Commands

| Command | Purpose |
|---|---|
| `credbrt init` | Create `config.toml` and a fresh database |
| `credbrt monitor` | Live terminal dashboard |
| `credbrt simulate [demo\|normal\|bruteforce\|spray\|lowandslow]` | Inject synthetic patterns |
| `credbrt ingest <file.jsonl>` | Tail a live auth-log file |
| `credbrt replay <file.jsonl>` | Bulk-load historical logs |
| `credbrt report --format [json\|csv]` | Export alert history |
| `credbrt reset` | Wipe the database |

## Configuration

All thresholds live in `config.toml`, generated by `credbrt init`:

```toml
[detection]
velocity_window_seconds = 60
velocity_fail_threshold = 8
stuffing_ip_fanout_threshold = 4
stuffing_account_fanin_threshold = 4
low_and_slow_window_seconds = 3600
low_and_slow_ip_diversity_threshold = 6

[scoring]
weight_velocity = 0.30
weight_stuffing = 0.30
weight_low_and_slow = 0.25
weight_fingerprint = 0.15
tier_medium = 30
tier_high = 60
tier_critical = 85

[alerting]
webhook_url = ""            # optional — POSTs a JSON alert payload
min_tier_to_alert = "MEDIUM"
```

```
├── detection/
│   ├── velocity.py           Sliding-window brute-force (per IP / account)
│   ├── credential_stuffing.py  Fan-out (spray) / fan-in (stuffing) detection

```

## Design notes

- **SQLite + WAL mode** so the dashboard can read while ingestion writes concurrently.
- **Detectors are pure functions of the DB state** — no shared mutable state between
  runs except the low-and-slow EWMA, which intentionally persists across scoring
  calls within a process to build up "heat" over time.
- **Simulator is a fixture generator, not an attack tool** — it writes synthetic
  rows directly to your local database, the same way you'd seed test data; it
  never sends network traffic anywhere.
- **Theme**: dark charcoal backdrop, Claude's coral (`#D97757`) as the primary
  accent, cyan (`#4FD1C5`) as a secondary "terminal hacker" accent, with an
  amber → orange-red → crimson severity ramp for MEDIUM → HIGH → CRITICAL.

## Extending it

- Swap `ip_reputation` table lookups into `credential_stuffing.py` to weight
  scores by a real threat-intel feed (AbuseIPDB, GreyNoise, etc.)
- Add a `geo_velocity.py` detector: impossible-travel detection using country/ASN
  deltas between consecutive successful logins for the same account.
- Add `--tail-app-log` adapters for specific stacks (Django, Rails, Okta System
  Log export) that convert their native format into the JSONL shape.
